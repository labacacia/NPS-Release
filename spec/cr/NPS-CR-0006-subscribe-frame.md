# NPS Change Request: SubscribeFrame (NWP §13) Formal Specification

**CR ID**: NPS-CR-0006
**Target version**: v1.0.0-alpha.11
**Status**: Accepted
**Accepted**: 2026-05-28
**Type**: Backward-compatible addition (new frame spec, new section in NWP, frame-registry status promotion)
**Author**: Ori, LabAcacia
**Affected components**: NWP spec (NPS-2) §13 (new), `spec/frame-registry.yaml` (SubscribeFrame `proposed → stable`), `spec/error-codes.md`, all 6 SDKs
**Issue**: GitHub NPS-Dev#44

---

## 1. Summary

`SubscribeFrame` (frame type `0x12`) was implemented in NPS-sdk-dotnet at alpha.8 as part of
the Anchor Node push-subscription infrastructure, but no corresponding spec section was
written at the time. The implementation shipped ahead of the spec to unblock SDK development.

This CR closes that debt by:
- Adding **NWP §13 SubscribeFrame** — field table, subscription lifecycle, heartbeat protocol,
  and cursor-based resume semantics
- Promoting the frame's status in `frame-registry.yaml` from `proposed → stable`
- Confirming that the five existing `NWP-SUBSCRIBE-*` error codes (already in
  `spec/error-codes.md`) are authoritative for this frame

---

## 2. Motivation

The alpha.8 SDK implementation has been in use since 2026-05-22. Several non-.NET SDKs are
being updated (alpha.11 parity sprint) and need a canonical spec to implement against.
Without a formal spec, each SDK risks diverging on field names, lifecycle semantics, and
error handling.

---

## 3. Spec Changes

### 3.1 NWP §13 — SubscribeFrame (0x12) (new section)

See the NWP v0.13 spec for the full field table, lifecycle description, and error codes.

Key normative rules added by this CR:

1. `subscription_id` MUST be a UUID v4 generated by the client; servers MUST echo it in
   all response events and the initial CapsFrame.
2. Server MUST respond to SubscribeFrame within the node's configured `DefaultTimeoutMs`
   with either a CapsFrame (`status = "open"`) or an ErrorFrame.
3. Heartbeat DiffFrames carry `event_type = "heartbeat"` and an empty payload; they do
   not increment any event counter toward `max_events`.
4. Cursor values are opaque server-generated strings; clients MUST treat them as
   non-parseable. An expired cursor MUST result in `NWP-SUBSCRIBE-SEQ-TOO-OLD` (not a
   silent resume from the beginning).
5. Server-side close (max_events reached, subscription expired, or error) MUST emit a
   terminal ErrorFrame with the appropriate `NWP-SUBSCRIBE-*` code before closing the
   stream.

### 3.2 frame-registry.yaml

Promote `SubscribeFrame (0x12)` status: `proposed → stable`.

No wire-format changes — the frame byte `0x12` and the field layout already shipped in
alpha.8. This CR only changes the registry metadata.

### 3.3 Error codes (no new codes)

The five existing codes are confirmed as authoritative:

| Code | HTTP | Meaning |
|------|------|---------|
| `NWP-SUBSCRIBE-STREAM-NOT-FOUND` | 404 | `subscription_id` unknown or already closed |
| `NWP-SUBSCRIBE-LIMIT-EXCEEDED` | 429 | Server's concurrent subscription limit reached |
| `NWP-SUBSCRIBE-FILTER-UNSUPPORTED` | 422 | Filter expression not supported by this node type |
| `NWP-SUBSCRIBE-INTERRUPTED` | 503 | Server-side interruption (restart, overload) |
| `NWP-SUBSCRIBE-SEQ-TOO-OLD` | 410 | Cursor position is no longer available in the log |
