# NPS-RFC-NNNN: {标题}

## 1. 摘要

_两到三句话。读者只读这一段就要能判断"这事跟我相关吗"。说清
**改什么**、**影响谁**——先不讲 **为什么**。_

## 2. 动机

_解决什么问题？为什么是现在？不做的代价是什么？尽量用具体数据——
实测延迟、实测体积、真实事故、真实用户诉求。_

_如果是响应外部压力（别家组织的规范、注册表政策、安全披露），给出
链接。_

## 3. 非目标

_明确不做什么。"本 RFC **不** 处理 X" 这种句子能省下几周的评审漂移。
相关但超出本 RFC 范围的问题，在这里声明，并可开一个占位 issue。_

## 4. 详细设计

### 4.1 Wire 格式 / 帧变更

_精确到 bit 的布局。字段表。编码字节示例。如果涉及
`frame-registry.yaml`，直接在这里贴 diff。_

```yaml
# frame-registry.yaml 示例 delta
- id: 0xNN
  name: NewFrame
  ...
```

### 4.2 Manifest / NWM 变更

_对 `/.nwm` schema 的增补。新字段必须声明是 required 还是 optional，
并说明不理解该字段的旧 Agent 的默认行为。_

### 4.3 错误码

_在 `spec/error-codes.md` 新增的条目。格式：
`{PROTOCOL}-{CATEGORY}-{DETAIL}`。每条必须映射到
`spec/status-codes.md` 里的一个 NPS 状态码；若需新状态码，在本节
声明并在同一个 PR 里更新 `status-codes.md`。_

### 4.4 状态机 / 流程

_变更涉及时序时给出时序图或 ASCII 流程图。包括超时、重试、
"成功"的判定条件。_

### 4.5 向后兼容性

_每项一句话——"是/否/不适用" + 理由：_

- 旧 Agent 能理解新帧吗？_（否 → 破坏性）_
- 旧 Node 能安全忽略新字段吗？_（取决于字段 required 还是 optional）_
- 版本不匹配时 on-the-wire 错误长什么样？
- 是否需要抬高 `min_agent_version`？_（是 → 21 天讨论窗口）_
